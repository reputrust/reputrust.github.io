Why GitHub stars don't measure trust

GitHub stars have become a vanity metric. Stars can be bought in bulk, inflated by bots, or driven by viral hype with no connection to code quality. A repository with 50,000 stars might have zero real users, while a repo with 500 stars powers thousands of production applications.

ReputTrust scores GitHub repositories on signals that actually matter: package downloads from npm, PyPI, crates.io, Docker Hub, NuGet, and Homebrew; issue resolution rates; contributor diversity; organic growth patterns; and CI/test coverage. These signals are hard to fake because they require genuine adoption and maintenance.

How the trust score works

Every repository gets a score from 0 to 100 based on five categories: real-world usage (are people actually installing and depending on this?), maintainer health (are issues being resolved?), community strength (is there more than one contributor?), growth pattern (is adoption organic or artificial?), and code quality (does it have tests and CI?). The score updates every 6 hours.

Supported package registries

ReputTrust checks downloads across npm, PyPI, crates.io, Go modules, Docker Hub, NuGet, Homebrew, and GitHub Releases. If a project is distributed through any of these, its real-world adoption is reflected in the score.
